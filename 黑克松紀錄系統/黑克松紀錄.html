我們的第一次 AI 驅動黑客松

從一個 PM 的點子，到一個 API 串接成功的原型。這是一趟關於速度、協作、意外發現與深刻學習的旅程。

我們為何舉辦黑客松？

最初的火花：從一個真實的痛點出發

  • 我們的起點並非為了比賽，而是為了解決一個真實的專案困境：一個需要耗費大量人力的紙本與 Excel 資料收集流程。
  • 最初的想法很單純：PM 能否利用 AI 工具快速產出一個前端原型，然後讓工程師快速串接真實 API，從而繞過緩慢的開發流程？
  • 我們想驗證：一個由 AI 驅動、跨職能合作的「一日衝刺」，是否能比傳統的開發模式更快解決問題？

意想不到的收穫：我們真正得到的價值

不僅是開發，更是「API 實境體檢」

  • 當團隊（尤其是後端工程師）實際操作一個需要使用 API 的真實場景時，才赫然發現部分既有 API 的設計，在真實情境下根本不敷使用。
  • 這次活動成了一場高效的「API 壓力測試」，讓我們在開發初期就修正了設計盲點，避免了日後更高的修改成本。

打破溝通的牆，建立共享情境

  • 一位工程師回饋，在活動前他「不太了解整個需求脈絡」。
  • 這次活動強制性地讓所有人「快速對焦、釐清、達成共識」，打破了資訊孤島，讓原本模糊的需求，成為了團隊共同的目標。

定義我們與 AI 的關係

  • 我們親身體驗了所謂的「介略問題」：AI 產出的原型雖然快，但難以直接用於持續性的產品開發。
  • 這次經驗幫助我們確立了重要的內部共識：AI MVP 是絕佳的「驗證工具」與「高保真需求文件」，而非「產品初版程式碼」。

點燃團隊的火花

  • 一位工程師說：「我反而覺得一開始給的需求『剛剛好』。」
  • 這代表活動創造了一個讓團隊能共同激盪、收斂想法的環境，每個人都是貢獻者，而不只是執行者。

我們的旅程：一次 MVP 之旅的核心洞察

完美的啟動：「剛剛好」的需求

  • 一位工程師回饋：「我反而覺得一開始給的需求『剛剛好』。」
  • PM 提供了清晰的目標，但保留了足夠的彈性，讓跨職能團隊能共同激盪出最佳方案，而非被細節綁死。
  • 此模式造就了活動初期「快速對焦、釐清、達成共識」的高峰體驗。

混亂中的轉機：API 深度體檢

  • 技能斷點：由於初期錯誤地只邀請了後端工程師，團隊一度因不熟悉前端框架而陷入停滯。
  • 設計盲點浮現：但也正是在這個混亂期，團隊透過實際操作原型，發現部分 API 在真實情境下根本不敷使用。
  • 敏捷的現場修正：我們當場就能討論並快速微調 API 設計，這遠比開發後期修改的成本低得多。
  • Hackathon 的新價值：它不僅是開發活動，更是一個絕佳的「API 實境壓力測試」機會。

具體的成果：一個真實可用的原型

  • 儘管過程曲折，最終成功在 Local Server 上跑起串接真實 API 的原型。
  • 此成果證明了整個想法的可行性。
  • 前端工程師證實 AI 產出的程式碼復用性高，可直接沿用，這是對 AI 價值的重要肯定。

隱藏的成本：超越程式碼的代價

  • 時間失控：活動嚴重超時，影響了團隊的精力與工作生活平衡。
  • 引入技術債：為求速成，採用了不安全的 CORS 臨時解法，需要在後續版本中償還。
  • 最後一哩路的挑戰：未能成功部署到測試環境，突顯了部署前置作業的重要性，也讓「成功」的定義變得相對模糊。

加速器：從 0 到 1，快得驚人

  • AI 在前端作業上能節省大量工時，讓 PM 能透過精準描述，快速生成可互動的 MVP。
  • 這使得團隊與客戶能跳過傳統的線框圖階段，直接在「真實」的介面上進行驗證與回饋。

整合挑戰：惱人的「介略問題」

  • 一次性產物：AI 產出的程式碼散落且缺乏架構，難以被打包出來，導致後續的迭代開發難以直接沿用。
  • 缺乏設計規範：AI 未能遵循公司預設的主題和顏色規範，需要額外手動調整。
  • 定位的轉變：我們學到，AI 產出的 MVP 更適合一次性的驗證，應將其定位為「最高保真度的需求文件」，而非「產品的初版程式碼」。

PM & Frontend 協作

這是一個專為 PM 與前端工程師設計的高效協作流程，旨在利用 AI 大幅加速從需求到可交付前端元件的過程，解決「介略問題」。

工作流程

PM 作業流程
  1. 發想功能需求：以單一功能為單位 (如：使用者註冊模組)。
  2. 使用 Gemini AI 產出程式碼：
    • 開啟 Gemini，複製 `GEMINI_PROMPT_TEMPLATE.md` 範本。
    • 根據功能描述，請 AI 產出 React 元件、型別定義及 mock data。
    • 確認產出格式正確。
  3. 貼入指定檔案：將程式碼暫存於 `libs/generated/<功能名稱>.tsx`。
Frontend Developer 導入流程
  1. 使用 AI 協助轉換：打開 Claude 或 Cursor，貼入 `AI_BUILDING_PROMPT.md` 指令，請 AI 協助重構。
  2. 拆解與搬移：AI 會將暫存檔自動拆解至正式專案結構（routes, types, mocks）。
  3. 整合與測試：實作對應路由，並完成最終整合與測試。

文件與 Prompt 設計

我們將 Prompt 標準化為文件，確保每次 AI 產出的品質與格式一致。

GEMINI_PROMPT.md (PM 使用)
# Gemini AI 專案程式碼產出標準 Prompt
請根據下列規格與要求，產出**完全符合專案結構**的 React Component、型別定義與 mock data，並分別以 markdown code block 輸出，且每段請加上檔名註解。

## 專案技術規格
- React 19 + TypeScript（strict mode）
- React Router v7（使用 routes 配置）
- TailwindCSS 4.x
- Vite 構建（不需考慮 Next.js）
- 型別需準確（interface）
- 專案使用 alias：`~/*` 代表 `./app/*`
- 輸出的程式碼必須用 markdown code block 包住，並加上檔名註解

## 專案結構要求
- React Component 請放在 `app/routes/.tsx`
- 型別定義請放在 `app/types/.ts`
- mock data 請放在 `app/mocks/.ts`
- 路由請以 `routes.ts` 配置，path 格式為 `/`

## 產出格式範例
`filename="app/routes/feature-name.tsx"`
// 這裡是 React Component 程式碼
...

額外注意事項
- 型別請盡量明確，避免 any
- 請勿產生與 Next.js 相關語法
- 請勿省略必要的 import
- 請確保所有檔案皆可直接複製貼上至專案對應路徑即可運作
- 若有多個元件/型別/mock，請分開多個 code block 並註明檔名
AI_BUILDING_PROMPT.md (Frontend 使用)
請幫我將以下这个檔案 `libs/generated/xxxxx.tsx` 的內容，整理成符合我目前專案的結構與邏輯的功能，請根據以下規範搬遷：

- 主元件請放到 `app/routes/xxx.tsx`
- 如果有型別請拆到 `app/types/xxx.ts`
- mock data 請拆到 `app/mocks/xxx.ts`
- 請幫我在 `app/routes.ts` 中新增對應的 route（以 `path: "/task-form"`）
- 請確保使用 TailwindCSS，並維持合理的 className 結構
- 不需要進行 fetch 或 API 串接
- 如果有可改善之處也請一起調整

我們走過的路：Hackathon 1.0 流程還原

這是在我們第一次活動中，實際發生的流程與各角色的任務。這份紀錄是我們優化流程的基礎。

活動前：點子與籌備

  • 探索AI潛力：認知到 AI 能將 Wireframe 提升為可互動原型，思考如何將此能力最大化，使其產出具備再利用價值。
  • 發起活動：基於驗證 AI 原型可用性的想法，產生了舉辦黑客松的點子。
  • 選定題目：選擇一個全新的「AI 排班」專案作為試驗田。（小發現：全新的內容更適合 AI，因不受限於現有框架，能讓 AI 最好地從 0 開始發揮。）
  • 事前溝通：活動前兩天，向團隊發出通知，提供主題、初步原型連結及相關資料 (Staff_files.xlsx)。
  • 主要處於被動接收邀請的狀態。
  • 對活動的具體需求與技術細節了解有限。

活動中：討論、執行與應變

第一階段：需求對焦與討論 (約 45 分鐘)

  • 【PM】主導說明需求、演示 AI 原型，引導團隊討論。
  • 【工程師】參與討論，確認共識與 Hackathon 目標，並在過程中發現既有 API 不敷使用的問題。

第二階段：實作、挑戰與救援

  • 【PM】根據討論結果，與 Gemini 進行多輪迭代 ( v1, v2, v3 )，即時修改前端程式碼；在發現開發停滯後，緊急尋求前端工程師支援。
  • 【後端工程師】著手修正先前討論發現的 API 問題；在後續整合階段，協助解決 API 格式、Auth、CORS 等問題。
  • 【前端工程師 (後期加入)】接手 Gemini 產出的 Code，將其調整為更符合團隊框架的模板，並將 Mockup 資料替換為真實 API。

我們前進的方向：Hackathon 2.0 優化劇本

這不僅是一份活動流程，更是一套從「發想到價值落地」的完整行動指南，將我們的學習融入其中。

階段一：籌備與共識 (活動前)

1. 點子與原型

  • 【PM】產生核心想法，用 AI 建立原型，為討論建立具體基礎。

2. 建立共識與定義

  • 【PM + 技術負責人】共同確立「雙軌開發」原則，明確 AI MVP 的定位是「快速實現的驗證工具」。
  • 【全員】定義 P0/P1 成功目標與「交接協議」(Handoff Protocol)。

3. 技術預檢與團隊

  • 【工程師】(技術負責人) 檢視原型技術棧，產出「必要技能清單」。
  • 【PM】根據清單，邀請「核心團隊」與「On-Call 專家」。

階段二：執行日 (Hackathon Day)

1. 啟動會議 (30分鐘)

  • 【PM】闡述目標，展示原型，強調「API 體檢」價值。
  • 【工程師】提問釐清，共同討論技術方案與分工。

2. 開發衝刺 (3小時)

  • 【PM】作為需求澄清者，並協助排除非技術障礙。
  • 【工程師】前後端配對或分頭編程，專注實作與整合。
  • 【專家顧問】(On-Call) 在團隊遇到特定難題 (如部署、CORS、設計) 時，提供快速支援。

3. 成果展示 (15分鐘)

  • 【全員】共同展示成果，分享過程中的亮點，慶祝成功。

階段三：覆盤與價值落地 (活動後)

1. Retro (30分鐘)

  • 【全員】坦誠回顧過程，總結學習點，不責備，只求進步。

2. 建立知識庫

  • 【全員】將有效的 Prompt 範例、技巧、心法整理至團隊共享的 **Prompt Playbook**。
  • 【PM】負責維護這份「黑客松學習歷程」文件，使其成為活的資產。

3. 啟動迭代

  • 【PM】將 Retro 中發現的技術債（如 CORS）和優化點（如 API 請求時機）轉化為正式的待辦事項。
  • 【工程師】認領相關任務，規劃進衝刺 (Sprint)，實現**小步快跑的持續改進**。
